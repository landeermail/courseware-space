# GitHub Issues 协作规则

GitHub Issues 是本项目重要开发任务的统一记录位置。两台开发电脑都以远端 Issue 为准，不使用某台电脑上的本地笔记作为任务事实来源。

## 何时必须创建 Issue

以下工作应先创建 Issue，再开始实现：

- 新增一道课件；
- 改变教学模型、动画逻辑或交互流程；
- 修正物理原理、公式、方向或数值关系；
- 涉及多个文件或多个课件的重构；
- 需要先讨论方案或明确验收标准的问题；
- 需要长期保留决策过程的任务。

以下小型改动不强制创建 Issue：

- 错别字和文案修正；
- 不改变教学逻辑的轻微样式调整；
- 范围明确的配置或文档修正；
- 可以在 PR 描述中完整说明的小型维护工作。

## Issue 应包含什么

重要任务的 Issue 应尽量说明：

1. 背景：学生目前难以理解什么；
2. 教学目标：课件应帮助学生建立什么直观认识；
3. 任务范围：本次要做和不做的内容；
4. 验收标准：物理正确性、动画表现、交互和目标视口要求；
5. 参考材料：题目、答案、截图或相关讨论。

不要求为了形式填写无关内容。

## Issue、分支与 Pull Request

- 每项较大任务使用独立分支开发；
- 有对应 Issue 时，在分支或 PR 中引用 Issue 编号；
- 完成任务的 PR 使用 `Closes #<编号>` 关联并关闭 Issue；
- 没有 Issue 的小型 PR，必须在描述中说明修改目的和验收方式；
- 不直接向 `main` 提交，正常变更通过 Pull Request 合并。

## Pull Request 不是需求入口

本项目目前由仓库所有者独立维护，不把外部 Pull Request 当作功能请求入口。新需求以 GitHub Issue 为准，PR 用于交付已经明确的任务。

## 常用命令

```bash
gh issue list --state open
gh issue view <编号> --comments
gh issue create --title "..." --body "..."
gh issue comment <编号> --body "..."
gh issue close <编号> --comment "..."
```

在仓库目录中执行时，`gh` 会根据 Git remote 自动识别仓库。
