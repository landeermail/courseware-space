# 领域知识文档规则

本项目采用单一领域上下文。稳定的项目知识记录在根目录 `CONTEXT.md`，重要且长期有效的决策记录在 `docs/adr/`。

## 开始任务前

在分析或修改课件之前：

1. 阅读根目录 `CONTEXT.md`；
2. 检查 `docs/adr/` 中是否有与当前任务相关的决策；
3. 使用领域文档已经确定的术语和原则；
4. 如果任务与现有 ADR 冲突，必须明确指出，不得静默覆盖。

如果这些文件尚不存在，可以继续工作，不要为了补齐目录而自动生成空洞内容。

## `CONTEXT.md` 记录什么

`CONTEXT.md` 用于保存跨课件、长期稳定的共同知识，例如：

- 项目的目标与边界；
- 互动课件区别于 PPT 的核心价值；
- 教学表现力和物理正确性的基本原则；
- 项目中的关键术语及其准确含义；
- 目标设备和主要使用场景；
- 明确不采用的产品方向。

它不用于记录单个任务的进度、临时方案或具体实现步骤，这些内容应留在 GitHub Issue 和 Pull Request 中。

## `docs/adr/` 记录什么

ADR 用于记录会长期约束未来开发的重要决策，例如：

- 全站架构或依赖策略；
- 课件模型和动画系统的共同设计；
- 跨课件复用方案；
- 部署、资源加载或兼容性策略；
- 曾有多个合理选项，且未来可能需要理解取舍原因的决定。

普通样式微调、局部修复和容易撤销的实现选择不需要 ADR。

## ADR 命名和内容

文件使用递增编号和小写英文名称：

```text
docs/adr/0001-example-decision.md
```

每份 ADR 至少说明：

- 状态；
- 背景；
- 决策；
- 主要理由；
- 后果与限制。

## 更新原则

领域知识应在真正形成共识时逐步补充，不预先编造完整体系。

当新任务澄清了一个稳定术语或共同原则时，更新 `CONTEXT.md`；当任务形成了重要且长期有效的取舍时，新增或替代 ADR。
